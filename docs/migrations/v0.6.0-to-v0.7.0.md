# Migration guide: v0.6.0 → v0.7.0

This release renames the project (`affinidi-webvh-service` →
`did-hosting-service`), introduces multi-domain hosting + multi-
method DID delivery, ships a companion client library, and adds
the canonical Trust-Tasks transport contract.

The on-disk store format is forward-compatible: a v0.7 binary
reads a v0.6 store directly and runs the M-01 migration on first
boot. **Forward-migrated stores cannot be downgraded** — see
`docs/rollback.md` for the recovery path.

## Pre-flight

1. **Take a backup.** Use the v0.6 backup tool or copy the data
   directory. The v0.7 daemon also runs `backup` with a v2 dump
   format that's forward-compatible with v1.
2. **Note the legacy `public_url`** from your `config.toml`. M-01
   uses it as the tier-2 fallback when a `DidRecord` has no
   embedded `did_id`.
3. **Audit ACL entries.** Existing `Owner` entries deserialise
   as `DomainScope::All` for backwards-compat. If you want
   per-domain scoping you'll narrow them post-migration.
4. **Stop every webvh-service binary** running against the
   target data directory. Multi-process write contention on the
   migration is not supported.

## Step 1: Upgrade the binaries

Pull the v0.7 release artefacts. Crate names changed:

```
webvh-server   → did-hosting-server
webvh-control  → did-hosting-control
webvh-daemon   → did-hosting-daemon
webvh-common   → did-hosting-common      (library only)
+ new          → did-hosting-client      (integrator-facing crate)
webvh-witness  → webvh-witness           (unchanged)
webvh-watcher  → webvh-watcher           (unchanged)
```

Binary names follow the crate names. Update any `systemd` unit
files, Docker `CMD`s, or wrapper scripts that reference the old
names. Env-var prefixes change from `WEBVH_*` to
`DID_HOSTING_*` for the renamed services (per-binary prefixes
like `DAEMON_*`, `CONTROL_*`, `WITNESS_*`, `WATCHER_*` are
unchanged).

## Step 2: Update the config file

Append a new `[hosting]` block to the existing `config.toml`:

```toml
[hosting]
# At least one domain MUST be configured. The first entry in this
# list becomes the system default — used by M-01's tier-2 fallback
# and by the ACL-default resolution rule (T34).
bootstrap_domains = ["example.com"]

# Grace period before unassignment purges DIDs. Default 2h.
# Format: "2h" / "30m" / "7d" / "45s".
unassigned_purge_grace = "2h"
```

If you operate behind a reverse proxy that rewrites `Host`,
also configure the proxy CIDR allow-list:

```toml
[server]
trusted_proxy_cidrs = ["10.0.0.0/8"]   # your LB / CDN ranges
```

Outside this CIDR set the daemon ignores `Forwarded` /
`X-Forwarded-Host` headers and uses the literal `Host`. Spoofed
forwarded headers from untrusted peers have no effect.

## Step 3: Start the daemon

The first boot runs three automated steps (no manual command):

1. **`seed_domains_first_boot`** — populates the `domains`
   keyspace from `bootstrap_domains`, or falls back to the
   legacy `public_url` host if you didn't set bootstrap. Loud
   warn-log if neither is configured.
2. **`seed_assignments_first_boot`** — same tier chain into the
   `assignments` keyspace. Ensures the server can host its own
   bootstrap domains before the control plane reaches it.
3. **M-01: `m01_tag_did_records_with_domain`** — walks every
   `DidRecord` and fills the new `domain` field. Tier 1: parse
   host out of the embedded `did_id`. Tier 2: system default.
   Records with neither (`did_id == None` AND no system
   default) are warn-logged and deferred — they'll be tagged
   on the next boot once a system default exists.

Verify the boot log shows all three steps complete:

```
INFO  first-boot domain seed         tier=bootstrap_domains count=1
INFO  first-boot assignment seed     tier=bootstrap_domains count=1
INFO  applying migration             migration_id=m01_tag_did_records_with_domain
INFO  M-01 complete                  tagged_from_did_id=N tagged_from_default=M
INFO  migration applied              migration_id=m01_tag_did_records_with_domain
```

## Step 4: Verify

Smoke-check the multi-domain enforcement:

```sh
# Resolve a known DID — must 200.
curl -sS "https://example.com/<mnemonic>/did.jsonl" | head -5

# Same DID, wrong domain header — must 404.
curl -sS -H "Host: other.example.com" \
     "https://example.com/<mnemonic>/did.jsonl" -o /dev/null -w "%{http_code}\n"
# Expected: 404
```

Admin surface:

```sh
# List configured domains.
curl -sS -H "Authorization: Bearer $TOKEN" "https://example.com/api/domains"

# Caller's scoped subset.
curl -sS -H "Authorization: Bearer $TOKEN" "https://example.com/api/me/domains"
```

## Step 5: Optional — tighten ACL scopes

Pre-existing `Owner` entries default to `DomainScope::All`. To
restrict an owner to specific domains:

```sh
curl -sS -X PUT -H "Authorization: Bearer $ADMIN_TOKEN" \
     -H "Content-Type: application/json" \
     -d '{"domains": {"kind": "allowed_with_default", "domains": ["example.com"], "default": "example.com"}}' \
     "https://example.com/api/acl/did:webvh:Q1:example.com:alice"
```

## Step 6: Upgrade integrators (clients)

If you have a Rust integrator that talked to the daemon, switch
to the new `did-hosting-client` crate (v0.1.0). The API is
documented in `docs/did-hosting-client-crate-spec.md`. Key
shape changes:

- Auth is unchanged on the wire (DIDComm v2 challenge-response).
- DID-management calls gain optional `method` + `domain` fields.
- Every authed REST call may carry a canonical `Trust-Task:`
  header — in v0.7 this is permissive, but new clients should
  always set it.

## What if M-01 fails

M-01 is idempotent. If the migration errors mid-walk (e.g.
unparseable record bytes), the applied-marker is **not**
written. The next daemon start re-runs the migration from the
top. Unparseable records are warn-logged and skipped — they
don't block the migration's success.

If you genuinely need to roll back, see `docs/rollback.md` —
the v0.6 binary cannot read v0.7-shape `DidRecord`s, so a
restore-from-backup is mandatory.
